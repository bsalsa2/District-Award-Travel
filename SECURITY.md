# SECURITY.md — District Award Travel

Last audit: 2026-06-09. This app stores real client PII: names, emails, phone
numbers, loyalty point balances, card *names* (never card numbers), elite
statuses, and travel dates. Treat every change against that fact.

---

## Current security posture

### Passwords
- **Storage:** bcrypt with per-password salt (`hash_pw` / `verify_pw` in `backend/main.py`). Never plaintext, never logged.
- Client access codes are generated by the admin at account creation and delivered out-of-band (email/text by the admin). They are bcrypt-hashed at rest.

### Sessions / tokens
- JWT (HS256), 7-day expiry, signed with `SECRET_KEY`.
- **Production refuses to boot** if `SECRET_KEY` or `ADMIN_PASSWORD` env vars are missing (no more insecure defaults — fixed 2026-06-09).
- Client portal stores its token in `sessionStorage` (cleared when the tab closes).
- Admin dashboard stores its token in `localStorage` (survives refresh — deliberate UX tradeoff for the single operator; revisit if more admins are added).

### Route protection
- Every `/api/admin/*` route requires a valid admin JWT (`require_admin`).
- `/api/client/me` (GET/PUT) requires a client JWT and only ever reads/writes the **token owner's** row — a client cannot reference another client.
- Unauthenticated surface is exactly: `GET /api/health`, `POST /api/intake`, the two login endpoints, and the static site files.

### Rate limiting (added 2026-06-09)
- `POST /api/admin/login` and `POST /api/auth/login`: **5 attempts / 15 min / IP**
- `POST /api/intake`: **10 / hour / IP** (each intake triggers outbound email, so this also throttles mail abuse)
- Enforced via `slowapi`; exceeding returns HTTP 429.

### Input validation
- `/api/intake` validates: well-formed email (no header-injection characters), every field length-capped (5,000 chars), max 120 keys per submission. Invalid → 422.
- Admin/client mutation endpoints use Pydantic models.
- All DB access goes through the SQLAlchemy ORM — no raw SQL, no string-built queries.

### XSS
- `client.html` and `admin.html` escape all user/admin-supplied values (`esc()`) before HTML interpolation.

### CORS
- Locked to the Render URL + production domains (`ALLOWED_ORIGINS` env var, comma-separated). Was `*` before 2026-06-09.

### Secrets
- All secrets live in Render environment variables (`render.yaml` marks them `sync: false` — never in git).
- EmailJS client-side mailer **removed** from the intake form (its public key was exposed in the page source); intake emails are sent by the backend over Gmail SMTP using `GMAIL_APP_PASSWORD`.

3. **Postgres backups:** Render **free-tier Postgres has NO automated backups** and the database is deleted if expired. Until upgraded:
   - Manual backup: Render dashboard → dat-db → "Connect" → run `pg_dump <connection-string> > backup-$(date +%F).sql` locally, weekly.
   - When revenue justifies it, the Render Starter Postgres plan adds daily backups. This is the single biggest data-loss risk in the system.

### Accepted risks / later
4. **Admin token in localStorage** — XSS-stealable in theory; mitigated by the esc() pass and single-operator use. Revisit with httpOnly cookies if a second admin is added.
5. **No CSRF tokens** — mitigated by bearer-token auth (no auth cookies) + locked CORS.
6. **Rate limits are per-IP in process memory** — resets on dyno restart; fine at this scale.
7. **Base64 screenshots inside the client JSON column** — growth risk, addressed in Phase 2 with client-side compression.
8. **No 2FA on admin login** — worth adding (TOTP) once the client book grows.
