# Security — District Award Travel

Last audit: 2026-06-09. Refreshed: 2026-06-13.

This app stores real client PII: names, emails, phone numbers, loyalty point balances, card *names* (never card numbers), elite statuses, and travel dates. Every change should be weighed against that fact.

---

## What's protected by default

### Passwords
Stored as **bcrypt** hashes with per-password salt (`hash_pw` / `verify_pw` in `backend/main.py`). Never plaintext, never logged. Client access codes are generated by the admin at account creation and delivered out-of-band; they are bcrypt-hashed at rest.

### Signing key — hard fail in production
`SECRET_KEY` and `ADMIN_PASSWORD` must be set as environment variables when `DATABASE_URL` is present (i.e., any Render deployment). If either is missing, **the process refuses to start** with a clear error message. There is no insecure fallback in production.

In local dev (no `DATABASE_URL`), a random 32-byte hex key is generated at process start. Tokens don't survive restarts — intentional.

### CORS
Cross-origin requests are restricted to an explicit allowlist. The default list covers:
- `https://district-award-travel.onrender.com`
- `https://districtawardtravel.com`
- `https://www.districtawardtravel.com`

Additional origins can be added via `ALLOWED_ORIGINS` (comma-separated). In local dev, `localhost:8000` and `127.0.0.1:8000` are added automatically. No wildcard origins.

### Rate limiting (slowapi, per-IP)
| Endpoint | Limit |
|---|---|
| `POST /api/admin/login` | 5 requests / 15 min |
| `POST /api/auth/login` | 5 requests / 15 min |
| `POST /api/intake` | 10 / hour |
| Other authenticated endpoints | 30–120 / min |

Exceeding a limit returns HTTP 429. Limits are in-process memory and reset on dyno restart — acceptable at this scale.

### JWT
Tokens are signed with **HS256** using `SECRET_KEY`. TTL is **7 days** (168 hours). Expired or tampered tokens return 401. The algorithm is pinned (`algorithms=["HS256"]`) — no algorithm-confusion attacks.

### Route protection
- Every `/api/admin/*` route requires a valid JWT with `role == "admin"` (`require_admin` dependency).
- `/api/client/me` (GET/PUT) requires a client JWT and only ever reads/writes the token owner's row — a client cannot reference another client's data.
- Unauthenticated surface: `GET /healthz`, `POST /api/intake`, the two login endpoints, `GET /api/public/*`, and the static site files.

### SSE token handling
The SSE stream (`/api/client/stream`) authenticates via a JWT in the query string because `EventSource` cannot set headers. The request-logging middleware logs only the path, never the query string — the token does not appear in server logs.

### XSS prevention
`client.html` and `admin.html` define an `esc()` helper that escapes `& < > " '` before any dynamic content is written via `innerHTML`. Static text uses `textContent`. The function is applied to every user-supplied or API-returned value before rendering.

### Honeypot
The intake form includes an invisible `hp` field. If the backend receives a non-empty `hp` value, the submission is **silently discarded** — returns `{"ok": true}` to avoid training bots to adapt. The discard is logged server-side at WARNING level.

### Input validation
Intake submissions are validated by a hand-rolled dict validator: well-formed email, all fields length-capped (5,000 chars each, 120 keys total). Invalid submissions return 422 before touching the database. Admin/client mutation endpoints use Pydantic models throughout. All database access goes through the SQLAlchemy ORM — no raw SQL, no string-built queries.

### Secrets not in git
All secrets live in Render environment variables. `render.yaml` marks them `sync: false` so they are never committed. The EmailJS client-side mailer was removed from the intake form (see below); intake emails are now sent by the backend over Gmail SMTP.

The EmailJS key (`d3KBln2jHflM1AYo5`, service `service_407s8wc`) was committed to git history in an earlier version of the repo. **Even though it has been removed from the current code, the commit history still contains it.** Anyone with read access to the repo can retrieve the key from `git log`.

Steps:
1. Log in to [emailjs.com](https://www.emailjs.com/)
2. Navigate to Account → API Keys
3. Revoke the old key and generate a new one
4. Update any remaining integrations (check if any other tooling still references the old key)

This is not optional. A leaked third-party API key can be used to send email on your behalf, incurring costs or triggering abuse reports.

### 2. Set a strong SECRET_KEY on Render

```bash
openssl rand -hex 32
```

Set the result as `SECRET_KEY` in the Render dashboard (Environment tab). Minimum 32 hex characters (256 bits). Rotating this key immediately invalidates all existing JWTs — all logged-in users must re-authenticate.

### 3. Set a strong ADMIN_PASSWORD on Render

Minimum 20 characters, randomly generated. Do not reuse a password from anywhere else. When `ADMIN_PASSWORD` is changed, the next server start re-hashes it and updates the admin row automatically.

### 4. Postgres backups — free tier has no automated backups

Render free-tier PostgreSQL has **no automated backups** and the database is deleted if the instance expires. A `DROP TABLE` or expired instance means the entire revenue ledger and client book are permanently lost.

Manual backup:
```bash
pg_dump "$DATABASE_URL" > backup_$(date +%Y%m%d).sql
```

The `.github/workflows/nightly-backup.yml` workflow runs `pg_dump` nightly, AES-256 encrypts the output, and stores it as a GitHub Actions artifact (7-day retention). Set up the `PROD_DATABASE_URL` and `BACKUP_PASSPHRASE` GitHub secrets to activate it. See `RUNBOOK.md` for the restore procedure.

When revenue justifies it, upgrading to the Render Starter Postgres plan adds daily automated backups. Until then, this is the single biggest data-loss risk in the system.

### 5. Set ALLOWED_ORIGINS in Render

```
ALLOWED_ORIGINS=https://district-award-travel.onrender.com
```

The default already includes this origin, but being explicit prevents confusion if the service is cloned or forked.

### 6. Set CRON_SECRET

If you use the `/api/cron/digest` endpoint for scheduled daily email digests, set a strong random `CRON_SECRET`:

```bash
openssl rand -hex 20
```

Without it, the endpoint returns 503. With a weak secret, anyone who can guess it can trigger digest emails.

## Accepted risks / deliberate tradeoffs

### Admin JWT in localStorage
The admin token is stored in `localStorage` rather than `sessionStorage`. This survives browser tab closes and page refreshes — deliberate UX for a single operator who asked not to be logged out on refresh.

Risk: an XSS vulnerability on the admin page could read `localStorage` and exfiltrate the token.

Mitigations in place:
- CORS is locked to specific origins; a script on another domain cannot make authenticated API requests
- `admin.html` is served from the same origin as the API
- All dynamic content in `admin.html` uses `esc()` before `innerHTML`

Accept or revisit: if a second admin is ever added, move to `HttpOnly` cookies with CSRF tokens.

### Client JWT in sessionStorage
Cleared on tab close. Clients must log in again per session. Accepted — clients access the portal infrequently (to check trip status), so re-login is a minor inconvenience with a meaningful benefit.

### No CSRF tokens
Bearer token authentication (Authorization header) is used, not cookies. CSRF attacks require the victim's browser to automatically send credentials — this happens with cookies but not with bearer tokens. The locked CORS policy adds a second layer. No CSRF protection needed for this auth model.

### No server-side JWT revocation
JWTs cannot be individually invalidated without a token blocklist (Redis, etc.). A stolen token remains valid for up to 7 days. Mitigation: rotating `SECRET_KEY` immediately invalidates all tokens across the system.

### Rate limits reset on restart
`slowapi` limits are in-process memory. A dyno restart resets all counters. Acceptable at current scale (one Render instance, handful of clients).

### No 2FA on admin login
Worth adding (TOTP) as the client book grows. Currently mitigated by rate limiting and bcrypt.

### SQLite in local development
Local dev uses SQLite when `DATABASE_URL` is unset. SQLite shares no data with the production Postgres instance. A small subset of SQL behavior differs between the two (certain constraint behaviors, JSON operators). The test suite runs against SQLite in CI.
